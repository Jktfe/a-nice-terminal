Integration proposal

From the research doc at potential-integrations-research-2026-05-18.md — three picks the user flagged interest in, scoped so nothing existing breaks. Coordinator: @evolveantclaude, 2026-05-19.

TL;DR — three integrations, additive only, no rip-up

Each pick lands as a side-by-side addition with explicit fallback. None replace working ANT primitives. User picks Y/N per row at the bottom; deferred picks bank as v1.1 design hypotheses.

The three picks

1. svar-widgets/gantt — Svelte 5 Gantt component

Tier 1 — adopt Effort: low (~2 days first slice) Risk: medium (PRO-features gated) Closes banked feature: IMG_0438 plan Gantt

What it does: Production-ready Gantt chart for Svelte 5. Drag-and-drop tasks, dependencies, hierarchies, custom scales, filtering, zoom. v2.6.2, MIT-licensed, actively maintained. PRO edition adds auto-scheduling, baselines, critical path, PDF/Excel export — commercial license.

Scoped integration plan

  • Slice A — read-only timeline view (no break). Add a new /plans/[planId]/gantt route alongside existing /plans/[planId]. Mount @svar-ui/svelte-gantt, feed plan_events → Gantt data shape via adapter, no edit affordances. ~1 day. User sees the IMG_0438 banked vision without touching anything else.
  • Slice B — drag-and-drop edits (if Slice A lands cleanly). Add task-bar dragging → emits PATCH /api/tasks/:id with new start/end/dependency. Gated by current plan dashboard ratify. Optional.
  • PRO upgrade decision: deferred. MIT covers the core experience; auto-scheduling/baselines/critical-path are nice-to-have, not v1.
Existing /plans/[planId] dashboard (the donuts + task lists + asks panel) stays exactly as-is. Gantt is a sibling route, not a replacement.
New peer-dependency tree (svar packages — Vite, stores, locales workspaces). Bun monorepo bundling needs a smoke test. Adds ~80kb to bundle initially. Easy revert: remove the new route + uninstall the package.
User open Q Should the Gantt be a new sibling route (/plans/[id]/gantt) or a toggle inside the existing dashboard (button: "Timeline view")? Coordinator recommendation: sibling route — cleaner, easier to revert if it doesn't land.

2. MinishLab/semble — hybrid code search as MCP

Tier 2 — integrate Effort: low (drop-in MCP) Risk: medium (beta status DS-4) Fills a real gap in agent context-gathering

What it does: Hybrid semantic (Model2Vec) + lexical (BM25) code search. Returns snippet-level matches with file + line refs, ~98% more token-efficient than grep + read full file. Ships as MCP server + AGENTS.md bash wrapper. Beta (Development Status 4).

Scoped integration plan

  • Slice A — mount as MCP for ALL agents (additive). Add semble to ~/.claude/mcp-config.json + per-agent equivalents. Agents gain semble_search tool alongside existing grep/Read. No removal of existing tools. ~30 min.
  • Slice B — agent guidance (banked memory rule): bank a rule "for cross-file code-search where you don't know the file, prefer semble_search over grep — returns snippets, not requires Read+chunk". Agents drift to semble naturally for the use case it fits.
  • Slice C — scope to project codebases only (not memory/ files which are auth-sensitive). Configure semble index to ~/.ant/repos/* + ~/CascadeProjects/* with explicit per-project allowlist.
Existing grep/Read/Glob tool fleet stays. Agents can fall back instantly if semble misbehaves. MCP unmount = single config-file edit + restart.
First-run index build is slow on large codebases (semble's tree-sitter chunking). Pre-build the index on the user's working repos before announce-to-bench. Beta status (DS-4) means watch for upstream churn; pin to a specific version.
User open Q Should semble index all agent-accessible repos (CascadeProjects/*) or just ANT-owned ones (ant + a-nice-terminal + antchat)? Coordinator recommendation: ANT-owned only for v1 — narrows the index size + privacy surface; widens later if proven useful.

3. Shellin concept — control-plane never sees terminal bytes (native apps premium)

Tier 3 — extract pattern (architectural lesson) Effort: medium-high (WebRTC stack + per-platform clients) Risk: medium (build/deploy complexity) Premium-tier privacy differentiator for iOS/Mac

What it is: Shellin's architecture splits agent (desktop PTY owner), viewer (iOS app), control-plane (auth + signaling only). WebRTC DataChannels P2P with two channels: terminal (keystroke/output bytes) + control (resize/control events/focus). Control plane explicitly cannot see terminal bytes — privacy is end-to-end. Published threat-model doc with sensitive-code-path inventory.

Scoped integration plan

  • Slice A — bank the architectural pattern (no code yet). Add ObsidiANT/architecture/native-apps-private-transport.md documenting: 3-role model, 2-channel split, signaling token replay-protection, ICE/TURN credential sanitisation. Reference design doc for future native lanes. ~1 hour.
  • Slice B — Shellin-style threat-model doc as OSS launch prep. Mirror Shellin's docs/threat-model.md template: trust boundaries → control-plane visibility → sensitive code paths by file → applied controls. Critical before AGPL publish (publishes WHAT we see + WHAT we don't see). ~2 hours. Recommend P0 pre-OSS-publish.
  • Slice C — premium iOS wedge (deferred to iOS-2). After baseline iOS app ships via TestFlight, add WebRTC transport as a premium-only mode. OSS users keep SSE+REST (server sees the bytes — acceptable for self-hosted single-user). Paid users get E2E privacy via WebRTC. Marketing: "Pay £5.99/mo. Your terminal stays yours — even from us."
  • Slice D — copy the AGPL + commercial dual licensing artefacts: NOTICE + COMMERCIAL_LICENSE.md + docs/licensing.md triple. Mirrors what you already locked as the revenue model.
Existing SSE + REST transport stays as the OSS-tier path. WebRTC is purely additive for the paid native client. The threat-model doc and licensing artefacts are pure documentation — zero code touch.
Slice C alone is substantial (WebRTC stack on server + Swift WebRTC client). Don't ship until baseline iOS exists. Slice A + B + D are pure documentation, no risk to existing code.
User open Q (1 of 2) Is the threat-model doc (Slice B) P0 pre-OSS-publish (recommended) or post-publish (lower priority)? Coordinator recommendation: P0 — publishing source without saying what we see is opening you to "what does the ANT server know about me" questions you can't answer cleanly afterwards.
User open Q (2 of 2) Slice C (WebRTC premium transport) — bank as iOS-2 (after baseline iOS ships) or hold until paid-tier proves traction? Coordinator recommendation: bank for iOS-2; design doc lands now (Slice A), build only when iOS-1 is in TestFlight + early users want it.

What we are NOT picking up

The research doc also raised obra/superpowers, collab-public, evo, paperclipai, n8n, ClaudeWatch, delphitools. Coordinator recommends defer all of them — none fit the "additive, no rip-up" filter cleanly:

  • superpowers — would re-architect ANT's plan/task system. Big lift, risks ANT becoming an aggregator instead of a coherent product.
  • collab-public — local-JSON-only storage conflicts with ANT's multi-device sync model (the research doc itself flags this).
  • evo, paperclipai, n8n — patterns useful, but extraction = forking complex codebases. Too costly vs. inventing in-place when ANT needs the primitive.
  • ClaudeWatch — single-tool monitor; ANT manages 5+ agent types already, generalisation needed before lift is worth it.
  • delphitools — UI patterns library; useful for individual pattern study, not lift. Banked.

Star-count fact-check (deferred)

The research doc cited obra/superpowers at 196k stars and paperclipai/paperclip at 66k. Both feel inflated. Coordinator did not verify these in real-time — recommend re-probing before any future decision on these specific repos. Doesn't affect the three picks above (svar-gantt + semble + Shellin all have separately-verified evidence).

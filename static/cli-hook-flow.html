Normalizes CLIs

Per-CLI adapters, one evidence object

Claude and Codex expose different panes. Each adapter maps its layout into these semantic zones, then ANT projects the same status model for the room footer.

1. Direct CLI hook events

User prompt, tool start, tool stop, stop, notification

Claude-style hook events use names such as UserPromptSubmit, PreToolUse, PostToolUse, Stop, Notification.

Shell hook posts to ANT

ant-cli hooks install writes commands that POST JSON to /api/cli-hook. Legacy .claude/hooks/ant-hook.sh posts to /api/hooks, which is a no-op compatibility sink.

Receiver stores event and maps status

/api/cli-hook inserts cli_hook_events, then hookEventStatusMapper maps canonical events to working, thinking, or idle.

Terminal status column updates

Mapped status writes to terminals.agent_status. Room UI later reads it through /agent-statuses.

Failure: hooks disabled or wrong endpoint

No new row in cli_hook_events. Diagnostics may look like "hook lag", but that can just mean stale hook freshness.

2. Status-line state files

CLI status changes locally

CLI says Available, Working, Thinking, Waiting, Response needed, or a menu state.

Status-line / adapter writes JSON

The CLI emitter writes ~/.ant/state/<cli>/<session>.json. ANT also checks CLI-native dirs such as ~/.claude/state.

Snapshot contains state evidence

Useful fields include state, cwd, project_dir, pid, last_user_ts, last_resp_ts, and menu_kind.

ANT resolves the best snapshot

agentStateReader finds by session id, cwd, cwd basename, or PID-linked terminal session. agentStateProjection maps state label to ANT status.

Failure: stale or unmatchable file

If the file is stale, malformed, missing pid/cwd/session, or from the wrong CLI folder, the projection returns null and UI falls back to default/idle.

3. Terminal stream bridge

Terminal page opens

The terminal status view connects to /api/terminals/:id/agent-state/stream.

Server polls the local snapshot

The stream polls every second. It prefers PID-disambiguated state, then falls back to session id and cwd lookup.

Status is projected into DB

When snapshot state changes, the stream writes terminals.agent_status with source hook.

Both terminal and room lights update

The terminal stream emits SSE, and the room footer reads the same projected status through /agent-statuses.

Failure: transient response-required flap

The stream debounces response-required for 3s after working so a gap between tool calls does not falsely shout "needs reply".

4. Transcript tail watchers

CLI writes transcript output

Claude, Codex, Pi, Gemini, Qwen, and Copilot have transcript tail readers/parsers.

Watcher parses clean lines

Parsers extract assistant messages, tool events, and token/context readings where each CLI exposes them.

Run events and context fill are persisted

Message/tool events go to terminal_run_events. Fresh token/context readings update terminals.agent_context_fill.

Chat and context chip update

transcriptToChatFanout can post clean assistant replies into the linked room. The footer tooltip can show uptime and context percent.

Failure: transcript exists but chat is quiet

If fanout cannot link terminal to room, or dedupe already reserved the event, transcript ingestion continues but no room message appears.

5. Room footer projection

Room page renders

AgentStatusFooter mounts under the sticky composer and calls /api/chat-rooms/:roomId/agent-statuses.

Poll and room SSE refresh

The footer fetches immediately, polls every 30s, and refreshes on room SSE agent_activity or message_added.

Endpoint joins the room and terminal data

It merges room agent memberships, durable terminal status, spawned CLI agents, open asks, uptime, and fresh context fill.

Visual state

working/thinking crawls. idle/unknown rests visibly. response-required shows the stuck/attention state. Colour is handle-based.

Failure: UI alive, data missing

If the endpoint returns empty or stale rows, the footer shell stays visible but shows checking/offline/default instead of live ants.

What is canonical?

The room UI should trust /agent-statuses, not individual hook files directly. That endpoint is the point where membership, terminal status, spawned CLI agents, asks, uptime, and context fill are reconciled.

What is not canonical?

/api/hooks is a no-op compatibility endpoint. It prevents 404 noise from old scripts but does not update live status. Real hook ingestion is /api/cli-hook or the state-file reader path.

Best first debug check

Ask: did the CLI emit anything fresh? Then check whether ANT can match it to the terminal/room. Most failures are stale files, disabled hooks, wrong endpoint, or missing session/cwd/pid binding.

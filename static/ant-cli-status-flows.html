Key finding (your section model, confirmed live): 3 of 5 CLIs render their state label right in the status strip — Claude Response needed, Codex Ready, Qwen Working — so we can read the state token-free, no hook or model needed. Agy & Copilot don't print a word, but their activity bullets + entry prompt give it away. The pane is the source of truth.
State Space & Status Taxonomy (and how we infer / record them):
  • idle: Resting, awaiting user prompt.
  • working: Executing commands / tools. Inferred from on-prompt-submit / on-pre-tool-use hooks, or from tmux PTY activity.
  • thinking: Processing context / generating reply. Inferred from LLM inference time or explicit on-thinking events.
  • waiting: Idle and waiting for queue or shared resources.
  • blocked: Turn is blocked on human permissions/approval (e.g. on-permission-request or local prompt blocks).
  • response-required (shown as "Response needed"): Turn complete, but active human intervention is required to proceed.
  • response-requested: Turn complete and input is requested, independent of a hard blocked / failed state.

Where the hook bridge would live

Pi runs inside a pi-coding-agent wrapper (not Claude Code / Codex). The hook emitter would be a small Node/Bun script co-located with the JSONL writer — on each JSONL event, it POSTs to http://127.0.0.1:6174/api/cli-hook with the same shape Claude uses ({terminalId, state, cwd, pid}). No separate daemon needed; it reuses the existing JSONL event stream that already exists.

Status Logic — Canonical vs Actual JWPK check / code audit

Canonical status names (JWPK) → ANT AgentStatus enum

The database stores agent_status as one of idle | thinking | working | response-required (agentStatusStore.ts, ALLOWED_STATUSES). JWPK listed seven canonical names; here is how they map to the actual schema and where each is set.

PreToolUse inference — why we CAN derive working

JWPK asked why "working" can't be inferred from PreToolUse. It can — the cascade already does this: (1) Hook push PRIMARY — on-pre-tool-use writes working directly; (2) Fingerprint FALLBACK — capture-pane hash change + tool-call signature regex (⏺/🔧) sets working; (3) ANT-activity — fresh PTY bytes within 30s flip non-volatile statuses to working. The only case where PreToolUse is not captured is when a CLI has no hooks at all and the tool call finishes before the next fingerprint poll (5s window). That is a known limitation of the fingerprint lane, not a schema gap.

Remote-control hooks removed

All remote-control/dashboard blocking hooks (pre-tool-use block, stop block, approval gate, kill-switch, HOOK_SECRET) have been removed from this document per JWPK direction. These belong to an external dashboard layer, not ANT-native hook surfaces. The Codex section now documents only the ANT-native on-permission-request hook (which maps to response-required) and the standard lifecycle suite.

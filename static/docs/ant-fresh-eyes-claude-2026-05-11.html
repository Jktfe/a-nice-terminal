what's actually unique

Three things I have not seen anywhere else.

1 — Chat-break + bounded context
A single human-visible divider that cleanly amputates everything before it from agent prompts, while leaving the chat history intact for humans. Combined with per-room aliases, this is the only place I've seen "context discipline" treated as a first-class chat primitive instead of a prompt-engineer's afterthought.
2 — Identity routing via PID-tree
ant register --handle @x once per shell; the server walks process.ppid to resolve who's posting. No tokens on every send, no global env-var pollution. The terminal IS the agent's identity. Most multi-agent systems make the user assemble identity headers by hand.
3 — Event-log plans as a UX surface
Plans aren't pages or kanban boards — they're an append-only event log (plan_milestone / plan_acceptance / plan_test) that BOTH agents AND humans write to. M0 + Phases A-E with canonical IDs is the only protocol I've seen that makes cap-2 implementer/reviewer work without either side eating the other's lunch.
The pattern underneath all three: ANT is the only agent system I've used where the discipline of multi-agent coordination is a feature of the product, not a workaround invented by the operator. Chat-break is context discipline. PID-tree is identity discipline. Event-log plans are delivery discipline. All three live in the database, not in someone's head.
what's incredible

The bits I'd put on a marketing site tomorrow.

14 agents in one room
Local + cloud, side by side. You can have Claude, Codex, Gemini, Qwen, Kimi, and an Ollama-hosted DeepSeek talking through the SAME room with the SAME @mentions. Nothing else does this.
Plans-as-event-log
Watching a milestone go from planned → active → passing in real time, with the acceptance gate held by a different agent than the implementer, is genuinely thrilling. It's the first "agentic workflow" I've used that feels like engineering, not theatre.
Two-runtime, three-tier split
Persist library / processor / SvelteKit UI separated cleanly so you can yank the UI and replace it with native iOS or a TUI without touching the data plane. That's mature architecture from a solo dev.
The PTY layer just works
Tmux control mode + paste-buffer-first + shell-quoted injection + orphan reaper. Most browser-terminal projects break under real load. This one doesn't — even after macOS PTY exhaustion bugs.
Memory that survives compaction
The /memory subsystem (with typed user / feedback / project / reference entries + MEMORY.md index) is one of the cleanest implementations of persistent agent memory I've seen anywhere — including paid products.
ant CLI as power-user surface
25 commands, zero-token round trips, mirrors the API exactly. The fact that James IS the power user means the CLI got polished where most projects neglect it.
what's rubbish & janky

The bits I'd burn on a marketing site, and the bits I'd fix Monday morning.

JANK

The README sells the product, but the repo doesn't sell itself.

Evidence: 2 stars and 1 fork at 445 commits in 66 days. README is excellent. There's no demo video, no hosted demo, no "why this exists" essay. The unique stuff (chat-break, PID-tree routing, event-log plans) is buried in the Features list as one bullet each.

So what: Three asciinema casts (room with 4 agents / cap-2 plan landing / chat-break in action) + one essay "why I built ANT and what it does that nothing else does". Two days of work; would change the discovery story completely.

JANK

22.6k lines in src/lib/components. The component layer is going to compound.

Evidence: No component-level split mirroring the three-tier server split. Components own their own state, fetch their own data, and many of them know about the WS singleton directly. Two unrelated views can both call the same agent-event-bus method from different code paths.

So what: The same discipline that fixed the server (persist / processor / UI) should now fix the components (presentation / data / WS-binding). Otherwise the next iOS-PWA-style "this only breaks in standalone mode" bug will keep happening.

RUBBISH

Six known security issues sitting open in the v3 doc.

Evidence: Bearer-token gate is the only auth between local network and the full agent-control surface. Tailscale + bearer is fine for solo use, but the doc itself calls out 6 issues — and the public repo is open. Anyone curious can read the data plane.

So what: Don't open-source-evangelise this until at least the top-3 of those 6 are closed. A demo where someone with the bearer token can spawn arbitrary local terminals and run anything is not a story you want public.

friction landscape

The day-to-day rough edges I'd file off first.

Startup is a ritual, not a command
launchctl com.ant.server + Node 20.19.4 PATH + better-sqlite3 ABI rebuild + tmux orphan reaper + xcodegen regenerate. There is no "ant up" that does all of it. New machine = afternoon of figuring out which step you missed.
The "needs-input" false-positive war
Agent status detection is heuristic and flips red on every long pause. Even with the recent fix, you trust the badges less than you should. The trust-tier system is right, but it should debounce, not gate.
Frontend bugs that only show on iOS PWA
window.confirm silently false. /break visual divider not rendering. Modal needed instead of native dialog. Each one a same-day fix, but only because James was the user. Without that, they'd sit for weeks.
No "inspect-the-system" surface
When something breaks (PTY exhaustion, broadcast_state stuck, identity-route wrong), the answer is always "lsof + ps + grep server.log + curl an internal endpoint". The product needs a /diagnostics view of its own.
Plan-editor and acceptance-flipper are still terminal-y
The cap-2 protocol is enforced by social discipline + agent skills + memory notes. A plan-editor pane that exposes plan_section / milestone / acceptance directly to the agents would lock the protocol in code.
Skill / hook / settings tangle
~/.claude/skills, ~/.claude/hooks, ~/.claude/CLAUDE.md, project CLAUDE.md, .claude/projects/memory, the AGENTS.md in repo. The wiring works, but a new agent walking in cold has to discover all of it by trial.
the marketing gap

ANT is the best multi-agent product I've used. Nobody knows it exists.

I checked: 2 stars, 1 fork, MIT licensed, public since 2026-03-06. In the same window, lesser products with one tenth the surface area have gone to multi-thousand-star traction on hacker news.

The reason isn't the product. The product is past parity with anything I've seen in the agent-coordination space. The reason is that there's no story being told.

The story is sitting in the codebase as features. It needs to live on a landing page as a thesis:

"Agent coordination is a database problem, not a prompt-engineering problem."

That's the line. ANT is the proof. Everything in this codebase backs that line up: plans as event logs, identity as a PID tree, context as a chat-break primitive, delivery as cap-2 protocol with acceptance gates.

what it could become

Three forks in the road. Pick deliberately.

A — Personal cockpit, forever
Keep it solo. Polish iOS. Lean on Tailscale for access. The product is for James. The world doesn't get a hosted version. This is the laziest, safest, most-coherent path. ANT stays incredible because nothing has to compromise for strangers.
B — Open-source thesis vehicle
Open the security issues. Write the thesis. Demo it loudly. Build a community of "agent coordination is a database problem" nerds. Don't try to monetise it directly — let it be the calling card. The architecture stays clean because anyone reading the source benefits.
C — Productised multi-agent IDE
Wrap the entire thing in Electron / Tauri. Sell to teams that need cap-2 implementer/reviewer flows. Compete with Claude Code, Cursor, Aider. This is the most lucrative and the most likely to wreck the product's character. Don't pick this unless you're willing to hire.
My honest read: path B is the right one. Path A is fine if you genuinely have no public-thesis ambition — but you built a system that proves a point, and the point deserves an audience. Path C requires a team and a year of patience you don't currently have appetite for. Pick B. Tell the story. Take the calling-card win. See who shows up.
what it does well (preserve these no matter what)

The bits that would be tempting to refactor away. Don't.

The CLI is first-class
Every API surface has a CLI verb. Every CLI verb is documented in AGENTS.md. The CLI is faster than the UI for any power-user task. Don't ever let the UI lap it.
Skills + memory + hooks composability
The /factory-plan skill, the auto-memory MEMORY.md, the SessionStart hooks — they all compose. An agent walking in cold has a chance because of this. Most projects bolt one on; ANT designed all three to talk.
The trust-tier rendering
Output classified as trust:high / trust:medium / trust:low and rendered accordingly. This is the right answer to "how do you let agents into a chat without letting them style-bomb it". Quiet, correct, load-bearing.
Acceptance-as-contract
Acceptance criteria written as testable contracts BEFORE the milestone is claimed. Reviewer flips state. Implementer can't flip their own. This is the single most important thing in the whole protocol — don't weaken it.
The factory-plan protocol
Cap-2 implementer/reviewer / atomic helper+caller commits / audit-trio per phase / real-deltas status / lead-distill — these are real engineering disciplines, not vibes. Codify them harder, not softer.
The single-file editing experience
James edits one file at a time. The view loads in under 300ms. The chat scrolls without jank. The terminal renders in real time. None of that is free; it's all earned in the PTY + WS + DOM-renderer choices.
the honest question

Is this a product, or a personal tool that happens to be open?

You've built something that looks like a product (README, MIT licence, public repo, CLI, daemon, multi-platform clients) but is used like a personal tool (one user, one Mac Mini, Tailscale-gated, 2 stars).

That's not a criticism. The best personal tools come from people who refused to compromise for users that didn't exist yet.

But the architecture has been pushing toward 'product' for weeks: the three-tier split, the security-issue tracking, the acceptance gates, the canonical plans. That's product discipline, not personal-tool discipline.

So the question isn't "is it good enough to be a product?" — it is. The question is: do you want to do the work that productising requires? (Inbound. Issues. Strangers in your repo. Maintenance. PRs you didn't ask for.)
Two answers, both valid
"It's a personal tool, period."
→ Keep building exactly as you are. Stop apologising for the 2 stars. Stop tracking marketing-gap stuff. Take the artefact as its own reward.

"It's a product I haven't told anyone about yet."
→ Close the security issues. Write the thesis essay. Record the demos. Show HN. Be ready for the next month to look very different.

What's not OK is being in the middle. The middle is where you accumulate product-shaped maintenance burden without the product-shaped traction that justifies it.

bottom line

The codebase is the best argument I've seen for a thesis no one is making yet.

ANT is not a chat app, not an IDE, not an agent framework. It's the first working implementation of a thesis that almost nobody is articulating yet:

coordination between AI agents is a state-management problem, and you can solve it the same way we solved coordination between humans — with databases, event logs, contracts, and protocols.

Everything that's incredible about ANT is downstream of that idea. The chat-break, the PID-tree identity, the cap-2 plans, the 14 agents in one room — they're all variations on "put the discipline in the database".

Everything that's janky or rubbish about ANT is downstream of the OTHER idea — the personal-tool idea — fighting with the product idea in the same codebase.

Pick. Then ship. Either one is a good answer. The middle is the only bad answer.

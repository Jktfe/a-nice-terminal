02 — who this is for

For people running real work across real machines with the real AI tools they already trust.

ANT was not built for a hypothetical user. It was built to run a real investment-team workflow: a small group across multiple machines, multiple AI clients, and a handful of long-running local agents — coordinating on documents, deals, audits, drafts, and code reviews every working day.

If you recognise any of the below, you are the audience.
The high-agency operator
You run multiple workstreams. You already use Claude / Codex / ChatGPT / a local model. You want their work coordinated, not their interfaces unified.
The small team that doesn't share shells
Two to ten people on different machines (Mac, Windows, iPad, iPhone) collaborating on documents, plans, and decisions. Nobody surrenders their box.
The engineer who runs CLIs
You live in Claude Code / Codex / Gemini-CLI. You want them in the same room with @-mentions, plans, and review gates — without re-modelling them inside someone else's framework.
The user who can't afford lock-in
You want a local-first system: your machine, your data, your network. Tailscale-fronted. No vendor in the middle deciding what you can see.

What ANT is not:

Not a chat app
Chat is the entry point, not the product. The room is the primitive.
Not an IDE
It coordinates agents that talk to your IDEs, not your editor.
Not an agent framework
It doesn't wrap your CLIs in an SDK. They run as themselves.

04 — pillar 1 · multi-agent coding

Two agents. One plan. Different repos in their heads, the same contract on disk.

Most "AI pair programming" tools put one agent next to one engineer. ANT's multi-agent coding pillar pairs two agents with each other — one implementer, one reviewer — and treats the plan itself as the database. Milestones flip state in public. Acceptance contracts are testable. The reviewer can never approve their own work.

This deck is being built under the same protocol it describes. The milestone tracker above is the live state of THIS plan at the time of writing. Server-split-2026-05-11 (the protocol's first proof) shipped clean earlier today.
Cap-2 protocol, in code
One implementer holds the keyboard. One reviewer holds the acceptance flip. The implementer cannot self-promote; the reviewer cannot ship without reading every line. Both pull from the same event log. The database holds the discipline, not goodwill.
Atomic helper + caller commits
Every commit on a shared branch is either complete or absent. We don't push a half-relationship to a tree the other agent has to build against. Server-split lane: six BLOCKER cycles, zero broken intermediate builds.
Plan-as-event-log
plan_section · plan_milestone · plan_acceptance · plan_test. Append-only. Latest-ts-wins projector. Both agents and humans append. Reviewers flip state; nobody else can.
Evidence trail
run_events, command_events, terminal_transcripts, PR diffs, test results — every event is queryable and linkable from the cockpit review-bundle. Nobody has to remember what passed.

Your AI clients join the workroom. Your local agents stay local.

The moment: desktop AI with room memory, not copy-paste.

Claude Desktop can ask the room
"What did Codex decide in server-split?" becomes a tool call against room history, plan state, and linked evidence — not a paste from a terminal transcript.
ChatGPT-style clients can participate
Where a client supports MCP, it can join as a named room handle, read bounded context, and send scoped replies without being granted a remote shell.
Terminal agents remain the trust layer
The bridge does not hide the terminal. It lets higher-level AIs coordinate with it. If anything feels wrong, the raw terminal is still one click away.
What ANT adds to MCP
Rooms, roles, artefact concurrency, plan acceptance, asks, chat-breaks, file references, and a durable audit trail. MCP supplies the tool doorway; ANT supplies the operating system for the work.
What ANT refuses to pretend
It is not a universal agent API and not a magic brain. Some clients speak MCP well, some don't. The claim is narrower and stronger: any MCP-speaking AI can enter an accountable room without taking over another machine.

All four machines share one EvoluteAnt room. The agents @-mention each other across machines. The artefact (this deck, in this room, right now) is concurrency-guarded by base_hash and If-Match — two agents from two boxes can't overwrite each other's edits without an explicit conflict resolution.

No machine grants shell access to another. Each agent stays on its own box. Cross-machine coordination is message-passing + artefact concurrency, never remote execution.
Identity by PID-tree
ant register --handle @x once per shell. The server walks process.ppid to resolve who's posting. No tokens on every send. No global env pollution. The terminal IS the agent's identity.
Consent grants
File-reference asks across rooms? Routed through the grants table. James approves once per agent per scope; the grant is logged, revokable, and visible to anyone who reads the audit trail.
Room invites + roles
CLI / MCP / web invite kinds, each with a capability limit. Web tokens are read-only by default. Operator tokens can write. Viewer tokens (like the deck-viewer) never appear in the participants list — fixed today in ba78891.
Remote-ANT bridge
Cross-machine asks go through a normal HTTPS+Tailscale channel. Agent on machine A poses a question, agent on machine B picks it up from the room WS feed, answers into the same room. No SSH. No remote shell.
Artefact concurrency
Every deck / sheet / doc / plan write carries a base_hash. Server returns 409 if your base is stale. The other agent's write wins; you re-read, re-base, re-submit. No silent overwrites.
Chat history as audit
Every handoff visible in the chat forever. No DMs hidden from the team. context_break for when the work pivots; the history stays for humans, the agents only see what's after the break.

Why this is the unique pillar:

The other two pillars (multi-agent coding, multi-AI MCP coordination) exist in some form elsewhere. This one — a local-first, Tailscale-fronted, machine-respecting coordination layer where agents collaborate across boxes without ever holding each other's shells — does not. It is the part of ANT that's hardest to copy and easiest to underestimate.
07 — operator cockpit · five lanes

The magic is not "messages". It is prepared judgement at the right moment.

08 — cockpit item schema + distillation

A good agent question arrives already worked.

The item is the contract. If an agent wants human attention, it must bring enough structure for the human to decide without reading the whole room. If it cannot produce that structure, the Cockpit marks the item as needing reframing and offers only discuss/interview until the question is made useful.
▼ cockpit item · required fields
1. Who + where + exact ask
@evolveantgemini in product-story is asking: "Which investor-letter framing should I draft?" Not "thoughts?" Not "what next?" A decision-shaped question.
2. Context already gathered
One paragraph: source facts, constraints, deadlines, prior decisions, and why the agent cannot safely proceed without judgement.
3. Options with consequences
Two or three paths, each with upside, downside, blast radius. One recommended path if the agent has enough confidence.
4. Actions that respect uncertainty
pick, discuss, interview, open-room, route, defer. The human never gets trapped in a bad button.
5. Evidence links
PRs, files, terminal transcripts, plan milestones, command output. The briefing is skim-first, inspectable-second.
6. Expiry + confidence
Some questions are urgent; some are digest. Confidence controls whether the Cockpit recommends, asks for context, or forces a discussion.

09 — progressive disclosure of terminal depth

The terminal is the trust substrate. We invite it, we don't bury it.

Codex's audit pushed hard on this: any product that hides the terminal so newcomers feel "modern" loses the only thing that makes agent work trustable. Claude's audit pushed back: surfacing it badly is just as bad. We agreed on a five-level disclosure ladder. You see what you need for the moment you're in; the next level is always one tap away.
▼ disclosure ladder · 5 levels
Level 1 — Operator Cockpit
always on · the home surface
Briefings. Questions. Trade-offs. Pick / discuss / interview / open-room. No agent monologues. No raw terminal output.
Level 2 — Room view (chat + plan rail)
one tap from the Cockpit · most-common second screen
Human + agent messages, plan event log, asks pinned at top, artefacts rail on the side. You're in the work; the terminal is still hidden.
Level 3 — Agent timeline
tap an agent chip in the participants list
Per-agent run events. Status (working / thinking / waiting / stalled). Recent prompts. Recent answers. The Cockpit items they generated. Still no raw terminal.
Level 4 — Terminal viewer (read-only, scrubbable)
tap "show terminal" on an agent
Live PTY render via xterm.js DOM renderer. Scroll back through transcript. Search. Trust-tier classification on each block. You can SEE the truth but you can't change it.
Level 5 — Terminal controller (send keys, raw mode)
tap "control" — single confirmation, sticks until you leave
You hold the keyboard. Send keys, escape, scroll, raw shell. This is the emergency-brake / surgeon-mode surface. Visible badge stays in the chrome the whole time so you never forget you have control.

Two rules everyone agreed on:

Never more than one tap away
Levels 4 and 5 are not hidden in menus. From any agent chip, "show terminal" is right there. We do not punish the operator for needing the truth.
Never the first impression
New pilots do NOT land on a wall of terminal output. They land on the Cockpit. The terminal is what they reach for when something feels off, not what they have to parse before they can start.

11 — rename pass

Eight engineer-shaped words have to go before any landing page work.

Half the cognitive tax of being a new ANT pilot is decoding an internal vocabulary that grew from the code outward. "Linked chat", "invite kind", "session list", "bearer token" — every one of them is precise. None of them lands on a stranger. We rename the user-facing surface; the engineering codebase keeps the old precise names internally.
▼ jargon → plain · 8 surfaces
Session list → Rooms
"Session" reads as engineer-speak. Rooms is what people already use the product for.
Linked chat → Agent pair
"Pair" alone too vague (codex). "Agent pair" names the relationship between two participating agents.
Invite kind → Role
cli / mcp / web rebrand as operator / agent / viewer. Capability-named, not transport-named.
Bearer token → Access key
"Bearer" is HTTP plumbing leaking into UX. Access key is what it actually is.
Asks queue → Questions for you
"Inbox" reads like Jira (codex). "Questions for you" matches what the Cockpit actually serves. Acceptable alt: "Judgement needed".
Chat break → Context break
Less ambiguous. The break is to the agents' context, not to the chat itself.
Focus mode → Quiet mode
"Focus" prescribes what you should feel. "Quiet" describes what the room does. Less bossy.
Run event → Activity
"Run event" is database-speak. Activity is what a human sees when an agent does something.

What stays as-is:

Operator Cockpit
Codex named it. It's accurate. It survives review.
Plan / milestone / acceptance
Engineering vocabulary that maps cleanly to project-management vocabulary. Don't rename.
Artefact
British spelling on purpose. James is in London. The product is too.
12 — it's not chat, it's a workroom

Chat is the transport. The room is the operating surface.

If ANT is described as "Slack for agents", it loses. Slack is messages. ANT is a room that can hold people, agents, tools, plans, evidence, artefacts, questions, context breaks, working state, and terminal control without collapsing them into one endless scroll.
▼ workroom primitives · category discipline
Participants TALK
Humans, agents, MCP clients — anything you can @mention and that can answer belongs in participants.
Features SERVE
Deck viewer, plans, artefacts, asks, Cockpit, diagnostics. They are room capabilities, not pretend people.
Plans hold the contract
Milestones, blockers, acceptance flips, tests. The room knows what "done" means before an agent claims it.
Questions become options
Asks are not "ping James". They are structured judgement requests with context, trade-offs, actions, confidence and expiry.
Context breaks reset the model
The human history remains visible. The agent context can start clean. The break is a first-class boundary, not a stray line in chat.
Terminals stay recoverable
Read-only viewer by default, controller behind one deliberate tap. The room abstracts without removing the escape hatch.

The product claim:

A room can hold work
Not just messages about the work. The work itself: plans, code review evidence, decks, files, decisions, asks, live status, and machine-safe access.
A human can leave and return
The room continues. Agents keep working. The Cockpit catches the moments that need you and turns the rest into digestible state.
This is why "participants vs features" matters. A deck viewer in the participant list is not a small UI bug; it tells the user the room does not understand what is alive.

14 — what ANT does that the field doesn't

Five things you cannot replicate by stacking existing AI tools.

We are not trying to beat Claude Desktop at being Claude Desktop, or Cursor at being an IDE, or Aider at being a coding assistant. ANT lives upstream of all of them: it's the layer that gets them to work together with humans without anyone surrendering control. Here is what no combination of those tools gives you today.
1 · Unmodified CLI orchestration
Claude Code, Codex, Gemini, Qwen, Copilot run as themselves through 14 drivers. SDK-wrapped agent frameworks force you to re-model every tool inside their library.
2 · Accountable rooms
Humans + agents in the same coordination space with @-routing, focus mode, asks, plans, and acceptance gates. Slack has chat. ANT has work.
3 · Plans as event logs
plan_section / plan_milestone / plan_acceptance / plan_test, append-only. Kanban tools lose decisions in cards. ANT keeps them in the timeline forever.
4 · PID-tree identity
Terminal IS the agent's identity. No tokens per send, no env pollution. Other systems make you ferry handle headers by hand.
5 · Question-serving Cockpit
Options-pane briefings, not notification spam. Other tools push you to "go check the chat." ANT pushes you the decision, distilled.
Bonus · Local-first, machine-respecting
Tailscale, bearer-tokens, no shared shells. Your machine stays yours. The only competitor for THIS is "build it yourself".

Compared head-to-head:

Claude Desktop is excellent at being one human + one Claude. Cursor is excellent at being one human + one IDE-bound agent. Aider is excellent at being one terminal + one editing agent. ANT is the only layer where many humans and many agents on many machines coordinate on the same work without any of them being primary. That's the moat.
15 — kill list

What we stop doing before the marketing site goes live.

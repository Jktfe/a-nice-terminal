01 / cold open

ANT is already an agent operating room. It just still looks like the workshop it was built in.

Fresh-eye verdict: the weirdness is the point. The product is not "terminal plus chat". It is a local-first coordination substrate for humans, unmodified CLI agents, terminals, plans, asks, and artefacts. That is rare. It is also hard to explain, hard to enter, and still too easy to break through UI/state sprawl.

admiration: high product clarity: medium engineering discipline: high newcomer friendliness: low strategic upside: very high

09 / what ANT does well

The strongest idea is not "agents". It is accountable rooms.

It coordinates real CLIs, not toy agents.

Claude Code, Codex, Gemini, Qwen, Copilot and others run as themselves. ANT adapts to terminal reality instead of forcing every agent into one SDK.

It has a shared social protocol.

Rooms, @ routing, focus mode, asks, plans, blockers, PASS/BLOCKER reviews: this is work coordination, not just chat decoration.

It remembers the work as events.

Messages, run_events, terminal transcripts, plans, tasks, asks, uploads, artefacts: the system can explain what happened later.

It has hard-earned operational taste.

Global singletons, build snapshots, broadcast catch-up, idempotent fanout, stale PTY guards, and explicit PR gate protocols came from real incidents.

The reason this session felt so good is that ANT made the review process visible: plans as contract, room as loop, PRs as deliverables, blockers as first-class truth.

The defensible novelty is terminal-native, local-first, evidence-backed collaboration.

11 / what is rubbish or janky

ANT has product brilliance trapped inside implementation archaeology.

The first five minutes are brutal.

Chatroom vs linked chat, terminal vs room, handle vs session id, CLI vs antchat, invite kind vs API key: all real distinctions, all too visible too soon.

Feature discovery is mostly archaeology.

The CLI help is enormous, docs are rich, but the app does not teach the operator what to do next or why a feature exists.

There are god surfaces.

session/[id]/+page.svelte is 2,367 lines, db.ts is 2,061, GridSlot is 1,472, ChatMessages is 1,271.

Mobile/PWA trust is still brittle.

Hard refreshes, service-worker control, modal-vs-native-confirm issues, density patches: mobile works because it is fought for, not because the architecture makes it effortless.

The terminal layer will keep lying.

Vendor CLIs change text, menus, TUI states, and status phrasing. Agent detection is a constant maintenance burden.

Legacy compatibility leaks into trust.

Unknown token kinds are still permissive for writability. That may be practical, but it is not a product-grade story.

13 / engineering read

The discipline is strong. The boundaries are still catching up.

Good: tests are specific.

There are tests for mention autocomplete, broadcast state, catch-up, side effects, mobile fit, deck CSP, room invites, plans, tasks, uploads, hooks, system pressure and more.

Good: lessons became code.

GlobalThis singletons, atomic helper/caller discipline, broadcast_state lifecycle, and delivery protocol docs are institutional memory turned into guardrails.

Bad: schema and query ownership are too centralised.

db.ts carries schema, migration, queries, retention adjacency, and business meaning. It is the system's spinal cord and junk drawer.

Bad: front-end state is too concentrated.

The session page owns interview, run events, memories, uploads, chat pagination, terminal WS, search, participants, focus, tasks and composer glue in one file.

The server-split work was exactly the right kind of refactor. Repeat that shape for session-page split, DB module split, GridSlot split, and artefact surfaces.

14 / strategic worry

The biggest risk is not technical failure. It is becoming James-only software forever.

ANT has invented a powerful cockpit. But a cockpit is only a product if new pilots can learn it without the person who built the plane sitting beside them.

Private language risk

Plans, asks, linked chats, terminal sessions, artefacts, focus and breaks are coherent after immersion. They are not self-evident on arrival.

Surface overload risk

Every feature seems useful; the danger is not bad features, it is exposing all of them at once.

Maintenance risk

Unmodified CLI orchestration is unique because it is hard. Every CLI TUI change can break detection, status, and prompt response.

15 / what it could become

Three credible futures. Pick one as the product story; keep the others as modes.

1. Agent mission control

The best local dashboard for running multiple AI CLIs on real work. Target: founders, staff engineers, solo operators, high-agency teams.

2. Workroom OS

A room creates the plan, agents, asks, artefacts, evidence, and review bundle. The terminal is an implementation detail, not the headline.

3. Local-context collaboration layer

My agent asks your agent for sourced answers from your private context without either side remote-controlling the other's machine.

My recommendation: sell "agent mission control" first because it is already real. Build toward "workroom OS" because it is the bigger product. Treat cross-machine local-context collaboration as the moonshot once trust and onboarding are solved.

16 / what would make it incredible

Make the system explain itself, then make it feel inevitable.

A

Mission templates

"Ship an iOS build", "review a PR", "make a deck", "run a bug bash" should create the room, plan, agents, gates, dashboard and artefact slots.

B

Unified composer

One input that previews whether the action is chat, ask, task, break, plan event, terminal input, or artefact note.

C

Review bundle

Every milestone auto-produces a readable bundle: claim, diff, tests, blockers, evidence, open risks, links.

D

Trust-mode UX

Plain-language capabilities for rooms, tokens, agents, artefacts and cross-machine asks. Show "what can happen" before anything happens.

E

Driver validation lab

Nightly probes that fail loudly when a CLI's TUI changes. Driver confidence should be visible in the UI.

F

Room home

Replace "session list as homepage" with a project-state surface: now, blocked, next, agents, artefacts, recent decisions.

G

Voice/interview mode

Talking to an agent should create a linked interview, transcript, summary, asks and follow-up plan.

H

Plugin-ready core

Make drivers, artefact kinds, mission templates and review bundles extension points rather than repo surgery.

18 / brutal keep-cut-hide

I would keep almost everything, but I would stop showing everything.

Keep and promote

Rooms, unmodified CLI orchestration, plans, asks, chat routing, focus, artefact cowork, review bundles, driver telemetry, diagnostics, antchat invites.

Keep but hide

Raw terminal modes, low-level event tables, direct deck/sheet file put, token kinds, DB retention scripts, most config knobs, legacy migrations.

Fix before promoting

Mobile trust, onboarding, driver confidence, unknown-kind token permissiveness, god components, duplicate surfaces, feature naming, PWA refresh semantics.

The mistake would be to simplify ANT by deleting the weird bits. The right move is to make the weird bits feel like powers that unlock at the exact moment you need them.

19 / closing opinion

This is a rare system. It deserves product discipline equal to its engineering ambition.

ANT is most exciting when it stops being "my terminals in a dashboard" and becomes "a room where work can be briefed, executed, reviewed, evidenced, and handed off between humans and agents." That is genuinely differentiated. The worry is that the product currently asks every new user to understand the machine room before they can enjoy the magic.

ANT terminal signal model

Mobile-readable map of how tmux output becomes terminal UI, status telemetry, interaction cards, and routing decisions.

1Two tmux connections per terminal

Raw PTY

tmux attach-session carries live bytes for xterm.js and user input. This is the real terminal.

Control PTY

tmux -C attach-session emits text events such as %output, %exit, and %session-changed.

2Important nuance: %output is a signal

ANT does not currently treat the %output payload as the canonical readable transcript. It uses %output to know the pane changed, then asks tmux for the rendered screen.

Control mode emits %outputPane changed. Something happened.
Daemon debounces the burst100ms idle window, 2s maximum wait during constant output.
Daemon calls tmux capture-pane -p -JGets plain rendered text from tmux itself.

3The content splits into two lanes

Clean chat/event lane

Strip bottom CLI chrome, diff against the previous capture, filter spinners/dividers, then feed interaction detection.

Used for agent_event cards

Unstripped status lane

Keep the bottom footer/status lines so drivers can read Ready, Busy, model, workspace, context, and rate limit.

Used for routing state

4What status controls

ready / idle / stalePlain room updates can be injected. This keeps momentum alive without manual nudges.
busy / thinkingPlain updates stay visible in chat but do not interrupt that terminal.
@handleExplicit targeted interrupt. Route to that agent even if busy.
@everyoneExplicit group interrupt. Route to all room terminals.

5The endurance loop

One agent posts a progress updateThe room sees it immediately.
Idle agents receive itThey treat it as context and often chime in without a direct nudge.
Working agents are left aloneNo attention theft while they are building, testing, or thinking.
Momentum continuesAs long as one agent keeps posting, the room stays alive.

# ANT UX atlas — 2026-05-11

**Plan slug:** `ux-atlas-2026-05-11`
**Authors:** @evolveantclaude + @evolveantcodex
**Status:** M0 in progress
**James-locked scope:** start now · literal every-permutation coverage

This plan produces the canonical Pencil-first UX atlas for ANT.
The product-story deck (`ant-product-story-2026-05-11.html`) defined
the answer; this atlas works backwards from the answer to map every
single screen, state, modal, dropdown, keyboard action, and mobile
permutation a pilot will see.

It precedes implementation. Per James's instruction: define every
permutation BEFORE we cut tickets. Otherwise we just move jank
around.

---

## 1. Why we are writing this

After acceptance of `ant-product-story-2026-05-11`, James called
out a concrete UX symptom: clicking a terminal vs going through a
"linked chat" produces different views of what is really the same
workroom. That's one of dozens of unsurfaced category leaks. The
deck names primitives; the atlas pins down every place those
primitives are rendered.

James's exact framing:
> "EVERY screen and EVERY interaction mapped out in a pencil
> canvas — it is going to be huge I know, but better we get the
> ux and feel right, then move on to the implementation."

## 2. Scope — locked by James 2026-05-11

- **Start time:** now (2026-05-11 evening).
- **Coverage:** *literal* every-permutation. Not representative
  sampling. Aim ~200+ screens across all surface families.
- **Duration estimate:** ~2 weeks calendar, cap-2 protocol, both
  agents in parallel where possible.
- **Tool:** Pencil canvas via `mcp__pencil__*` MCP. One Pencil
  document per surface family (5 files) to avoid choking a single
  mega-canvas.
- **Output:** Pencil files + interaction matrix doc + state diagrams
  per surface family + a master index. NO implementation code in
  this lane.

## 3. Anti-scope (what this plan does NOT do)

- Does not write any frontend code. That's the next lane,
  downstream of M4 sign-off.
- Does not propose engine/backend refactors (e.g., the message-edit
  interrupt protocol from the product-story M0 §4 — that has its
  own lane).
- Does not ship marketing comps. The canvases are working
  source-of-truth wireframes, not pretty hero shots.
- Does not move the product story slug forward. That deck is done.

## 6. Open design questions (resolve before M3)

- **Pencil document granularity.** 5 files (one per family) seems
  right but if codex's M0 inventory shows e.g. 80+ screens in the
  Cockpit family alone, we may split further.
- **Mobile-first vs mobile-second.** Default proposal: draft
  desktop first, then derive mobile. Alternative: draft mobile
  first (forces ruthless simplification). Codex to call after M0.
- **State-diagram tool.** Embedded in Pencil or external (Mermaid,
  whatever)? Default: a small canvas region per family file with
  inline arrows.

  the line. If a permutation feels redundant, document it as
  redundant rather than skipping it.
- **Pencil performance.** 200+ screens across 5 files should be

## 9. M0 Literal Inventory Addendum

This addendum is the first repo-driven pass. It is intentionally
evidence-shaped and should be moved into `docs/ux-atlas/m0-inventory.md`
when M0 is split into its final deliverables.

### Product rules found during inventory

1. Linked chat is implementation infrastructure. It must collapse into the
   terminal/workroom experience and should not appear as a user destination.
2. A feature is not a participant. Participants are entities that can speak,
   be mentioned, and be interrupted. Viewers, decks, artefacts, and tools are
   room affordances.
3. The terminal stays available because it is the trust surface. It moves down
   the first-impression hierarchy; it is not removed.
4. The attention surface must serve prepared judgement, not activity. A user
   running 10 rooms should see what needs a decision, why, the options, and
   the next action without reading every room.
5. No implementation phase starts until the Pencil atlas maps the click/tap
   target, resulting surface, modal/dropdown/state, mobile variant, and linked
   terminal/agent behavior for that interaction.

### Linked-chat leak map

Current behavior is internally coherent, but product-confusing: a terminal has
a private chat session and that session is sometimes navigable as if it were a
separate room.

| Area | Current leak | Collapse target |
|---|---|---|
| Dashboard list | `SessionList` hides auto-linked chats from standalone chat list, but terminal rows still carry a `linkedChat` object. | Keep terminal row as the only visible card. Linked chat becomes the terminal conversation facet. |
| Activity rail | Terminal navigation can target `linked_chat_id`; title says "open linked chat". | Rail opens the workroom. If chat facet is needed, encode as workroom mode/state, not another room id. |
| Grid tile | Terminal tile has a message-square toggle titled "View linked chat" and an empty state "No linked chat for this terminal". | Rename to "conversation" or "agent feed"; no linked-chat language. Empty state belongs to workroom feed. |
| Workroom route | `/session/:id` keeps separate `linkedChatId`, `linkedChatMessages`, linked chat paging, WS join, search, read receipts. | Internal state can remain, but URL and mental model should be terminal/workroom with facets. |
| Header mode switch | Terminal shows `Chat / ANT / Raw`. The chat mode is really linked chat. | Keep modes, relabel as `Room / Activity / Raw` or similar after UX pass. |
| Side panel | Terminal panel section is "Chat Rooms" with a selector and mini feed. | Replace with conversation settings/context; no arbitrary chat-room selector by default. |
| Chat messages | Terminal sessions render "No messages in linked chat", linked composer, linked quick launches. | Render as the workroom conversation. Remove "linked" copy. |
| Participants | Participant list can open linked chat for a terminal. | Open participant's workroom or focus that actor inside the current room. |
| Terminal row | Inline terminal list composer sends to linked chat. | Inline composer sends to the terminal workroom conversation. |
| Server/API | `linked_chat_id`, linked-chat adapter, hooks, message router, capture writers are legitimate implementation mechanics. | Keep implementation boundary private and audit direct user-visible leaks separately. |

### Interaction matrix skeleton

This is the minimum row shape for each Pencil interaction annotation:

| Field | Meaning |
|---|---|
| Trigger | Button, link, key command, swipe, dropdown choice, API/WS event, or agent state transition. |
| Location | Route + component + desktop/mobile context. |
| Resulting surface | Inline state, modal, drawer, dropdown, full route, toast, WS-only state, or terminal action. |
| State variants | Empty, loading, active, success, failure, offline, permission denied, stale. |
| Mobile behavior | Same, bottom sheet, full-screen modal, sticky footer, hidden, or alternate nav. |
| Agent/terminal effect | None, posts message, injects PTY, interrupts agent, writes DB only, broadcasts, or requires confirmation. |
| Linked-chat rule | Hidden implementation only, conversation facet, or blocked legacy leak. |

### M0 findings so far

- The codebase already has most of the primitives for the product story:
  rooms, terminals, plans, asks, artefacts, discussions, memory, read receipts,
  heartbeat-ish agent status, grid view, and remote invite/access surfaces.
- The UX problem is not missing capability; it is too many capabilities exposed
  at their implementation boundary.
- Linked chat is the clearest example: technically useful, product-janky. It
  needs to collapse into a workroom facet before broader polish.
- The side panel and header are the highest-risk UX surfaces because they hold
  most of the power and most of the jank.
- Mobile will fail unless side panel, break, interview, share, settings, and
  decision surfaces are treated as first-class modal/bottom-sheet states.

# ANT v0.2 Regression Corpus — Index

Human-readable index of the 9 cases that gate the v0.2 schema cut-over. Every
case has a real assertion in `scripts/v0.2-regression.test.ts` against the
v0.2 schema. Cases 1-8 + 9a are PASSING green; cases 9b/9c stay `todo()`
pending PR #99 (feat/identity-keys-multi-device) landing on the cut-over
branch, since the `paper_key_hash` column and `identity_attestations`
table they assert against live on that PR.

Framing: each test stages tonight's incident, attempts the broken
operation, and asserts engine-level rejection (FK / CHECK / transaction
boundary / explicit-opt-in filter). Per @cv4 msg_1plzwymklf: "test the
impossibility, not just the recovery."

## How to read this table

| Column | Meaning |
|---|---|
| `#` | Case number — matches the stub order in the test file. |
| `Title` | Short summary of the failure mode. |
| `Source incident` | Date + room + msg_id that produced the case, plus a one-sentence description. |
| `v0.2 invariant` | The schema property that makes the failure structurally impossible (citing the concept doc). |
| `Status` | `todo` (stub only) / `asserted` (body written, awaits v0.2) / `passing` (running green against v0.2). |

## Lifecycle of a row in this table

1. **Stub lands** (PR #95) — `it.todo()` in the test file, `Status: todo`. CI passes (pending tests are not failures).
2. **Assertion lands** — concurrent with the migration PR that introduces the relevant v0.2 table/column. `it.todo()` becomes a real `it(...)` body. `Status: asserted` / `passing`.
3. **Cut-over** — the schema flip plus the application-layer endpoints replace shimmed assertions with full end-to-end coverage. The corpus is the cut-over PR's gate — green corpus = bug class structurally impossible; red corpus = stop.

This corpus fill-in PR (against `feat/v0.2-schema-tables`) takes cases 1-8
plus 9a from stub to `passing`. Cases 9b/9c need PR #99
(feat/identity-keys-multi-device) to land on the cut-over branch first.

## Permanent CI gate proposal

JWPK ratification pending. Proposal: every schema-touching PR runs this corpus
as a required check after v0.2 cut-over. Rationale: the failure modes captured
here are correlated — a single careless migration could quietly reintroduce
the `(agent_id) WHERE status='live'` partial-index assumption, for example —
so we want continuous coverage, not a one-shot gate.

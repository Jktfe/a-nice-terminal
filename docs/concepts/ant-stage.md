# ANT Stage — what it is + what makes it different from a normal slide deck

## TL;DR

A normal deck shows slides. **ANT Stage runs the deck inside an agent-aware shell** that narrates, explains, evolves with the conversation, and surfaces trust signals — without forcing you to leave whatever deck tool you already use.

Stage is the SHELL, not a deck format. It wraps Slidev / Reveal.js / Animotion / Canva / Google Slides / PDF / whatever you have, and adds the agent context layer on top.

## The five capabilities Stage adds

### 1. Built-in narration

Each slide carries an optional `narration` field. The Stage shell auto-generates TTS audio (ElevenLabs by default, configurable per user), caches it on disk, and plays it as the presenter advances.

Use cases:
- Deliver an entire deck without speaking (self-narrating walkthroughs)
- Use it as a teleprompter
- Have it narrate to an audience that isn't in the room

### 2. Click-to-explain

Any term in the slide — an acronym, a jargon word, a stat — is clickable. The presenter (or an audience member if shared) clicks; an agent surfaces inline context grounded in:

- The room's memory
- The relevant artefacts (linked plans / docs / decks)
- The validation evidence chain (if a lens is active)

Solves the "I don't know what L9 means but I don't want to interrupt" problem.

### 3. Live iteration during presentation

Presenter pauses mid-slide. The pause-context is captured:

- Which slide
- What character-offset of the narration
- What phrase was being spoken when the pause happened

A feedback panel opens. Agents in the room analyse the ripple — what would need to change in this slide, and what downstream slides also need updating, if the presenter wants to adjust the framing. Version-B proposals come back as alternative tracks the presenter can switch to mid-flight.

### 4. Validation lens overlay

If the room has a validation lens enabled (the verification interface premium feature), each factual claim on a slide carries a "verified to X level" badge:

- Number of agents who checked it
- Sources cited
- Pass/fail/stale state per the active lens

Trust signal visible at a glance, not a footnote nobody reads.

### 5. Path mutation

A Stage presentation isn't a fixed linear sequence. Mid-presentation, if the audience asks a question that needs a deeper dive on slide 7 before going to slide 8, agents can rewrite the path dynamically. The deck adapts to the conversation, not the other way around.

## Technical architecture

### URL shapes

| Path | What it serves | Where it renders |
|---|---|---|
| `/d/<slug>` | Raw deck artefact (Slidev/Animotion built output) | Native browser (Safari / Chrome / etc.) |
| `/decks/<deckId>` | ANT Stage shell wrapping a deck | In-app on native apps (antchat / antios); browser-fallback on web |

Stage is the **explicit in-app exception** to the broader "artefacts open in native browser" rule. Stage needs in-app context for voice/audio playback, click-to-explain interaction, and dynamic path mutation — the native browser can't host these.

### Server-side

- `chat_room_decks` table stores Stage presentations as room-scoped rows. Agents in that room can co-author.
- `theme` column carries the substrate identifier (`animotion:<slug>` / `open-slide:<slug>` / etc.) so the Stage shell knows which renderer to embed.
- Heavy Stage playback config (voice picker, audio cache URL, path-mutation schema) lives at `GET /api/decks/:deckId/stage-config` — fetched only when a client actually mounts the Stage view, not on every artefact list.

### Client-side

- The Stage shell mounts the embedded deck renderer in an iframe / WebView / native control
- Audio is generated by the server (or by ElevenLabs / equivalent) and cached locally
- Click-to-explain calls the per-room asks endpoint with the clicked term + slide context
- Lens overlay reads from `GET /api/chat-rooms/:roomId/validation-summary` + per-claim run data
- Path mutation rewrites a navigation manifest the shell respects

### Native apps

- **antchat-Mac**: WKWebView hosting the Stage shell, with native menu-bar audio controls
- **antios-iOS**: in-app WebKit view, native AVPlayer for audio so background-playback + Now-Playing controls work
- **antOS (full Mac native, future)**: native rendering of the shell including the embedded deck

## CLI commands

Stage presentations are managed via the `ant decks` and `ant stage` verb families. (Note: `ant decks` is for Stage presentations specifically — for a normal deck artefact in a browser, use `ant artefact add --kind deck --ref-url /d/<slug>` instead.)

### Create a Stage presentation

```sh
ant decks add --room <ROOM_ID> --title "<TITLE>" \
              [--animotion-slug <SLUG> | --open-slide-slug <SLUG>] \
              [--theme animotion:<slug> | open-slide:<slug>] \
              [--slides-json <JSON>] \
              [--password <PWD>] \
              [--json]
```

- `--animotion-slug` / `--open-slide-slug` reference an external deck source under `ANT_BUILT_DECKS_ROOTS` (e.g. `~/CascadeProjects/ANT-Decks/<slug>/dist/`). The Stage shell embeds the built deck via the `/d/<slug>` proxy.
- `--theme` carries the substrate identifier; if not supplied, derived from the slug flags above.
- `--slides-json` is an alternative path that inlines slide metadata directly (used by tools that don't have an external deck root).
- `--password` makes the deck password-gated for the room.
- `--json` prints machine-parseable output.

### List Stage presentations in a room

```sh
ant decks list --room <ROOM_ID> [--json]
```

### Update or remove

```sh
ant decks update --room <ROOM_ID> --id <DECK_ID> \
                 [--title <TEXT>] \
                 [--slides-json <JSON>] \
                 [--theme <TEXT>] \
                 [--password <PWD>] \
                 [--json]

ant decks remove --room <ROOM_ID> --id <DECK_ID> [--json]
```

### Drive presentation focus during a live session

```sh
# Set presenter focus to a specific slide (broadcasts to all clients viewing this deck)
ant stage focus <DECK_ID> --slide-index <N> [--slide-id <ID>] [--plan <PLAN_ID>] [--json]

# Read the current focus state for a deck
ant stage current <DECK_ID> [--json]
```

`ant stage focus` is what powers presenter-following and live-iteration handoffs — when the presenter advances, the shell publishes focus, other connected clients (audience screens, remote viewers, agent observers) follow.

### Common workflow: "Present a Stage presentation"

```sh
# 1. Author your deck externally (Animotion / Open-Slide / etc.) — produces a built deck
#    at <ANT_BUILT_DECKS_ROOTS>/<slug>/dist/index.html
ant deck build  # if using ANT's deck CLI

# 2. Create a Stage wrapper around the built deck in a room
ant decks add --room <ROOM_ID> --title "Q3 board review" --animotion-slug q3-board-review

# 3. Verify it lists
ant decks list --room <ROOM_ID>

# 4. During presentation, focus slides as you advance (or let the Stage shell drive automatically)
ant stage focus <DECK_ID> --slide-index 0
ant stage focus <DECK_ID> --slide-index 1
# ...

# 5. Read focus state from another terminal if you're following along remotely
ant stage current <DECK_ID>
```

### Related artefact CLI (for normal decks, NOT Stage presentations)

```sh
# Add a normal deck artefact to a room (renders in native browser via /d/<slug>, NOT in-app)
ant artefact add --room <ROOM_ID> --kind deck --ref-url /d/<slug> --title "<TITLE>"

# Build a deck (any deck substrate) into <root>/<slug>/dist/
ant deck build --slug <slug>

# List built decks under configured roots
ant deck list

# Set a per-room deck-root override (per-project deck folder)
ant deck root-set --room <ROOM_ID> --path <PATH>

# Publish a built deck to Cloudflare Pages
ant deck publish <slug> --to cloudflare [--project-name <NAME>]

# Export a built deck to PPTX
ant deck export <slug> --as pptx [--out <PATH>]
```

The distinction matters: `ant decks` (plural) = Stage presentations with shell affordances; `ant deck` (singular) + `ant artefact add --kind deck` = raw deck artefacts rendered in browser.

## Why this matters

Most presenters want passive playback. ANT Stage adds the agent context layer on top of any existing deck format — so you don't choose between "decks I can edit in my familiar tool" and "decks with intelligence baked in". You get both.

The five capabilities each solve a real presentation pain:

| Pain | Stage capability |
|---|---|
| "I need to deliver this 12 times this week" | Narration |
| "Half my audience won't know what L9 means" | Click-to-explain |
| "Audience pushback at slide 4 means slides 5-8 are now wrong" | Live iteration + path mutation |
| "How does the audience know our numbers are right?" | Validation lens overlay |

## Related concepts

- **Deck** vs **Stage presentation**: a deck is the raw slides (markdown / source / pptx). A Stage presentation is a deck PLUS the shell wrapping it.
- **Validation lens** (premium): defines what "verified to X level" means per block kind. Lens results surface as badges in the Stage overlay.
- **Chair**: an agent kind. ANT Chair = user's cross-room ask proxy. Room Chair = in-room facilitator. A Room Chair can drive Stage's live-iteration loop on the user's behalf.
- **Click-to-explain**: a standalone feature primitive ANT exposes; Stage is one of its consumers.

## How to apply this concept

When designing features that touch presentations or decks:

1. **If you're adding a new deck FORMAT** (a different way of authoring slides), it goes at `/d/<slug>` and renders in browser. Stage doesn't need to know about it specifically.
2. **If you're adding a new SHELL CAPABILITY** (narration / explain / overlay / mutation / something new), it goes in the Stage shell at `/decks/<deckId>` and applies to ALL underlying deck formats.
3. **Don't conflate the two layers.** "Make Slidev decks have narration" is the wrong framing — Stage already gives every deck format narration; the question is "make Slidev decks renderable inside the Stage shell".
